

> [!important] 정리
> **Spring은 CheckedException을 '예상된, 복구 가능한 상황'으로 봐서 롤백 안 함.**  
반대로, **UncheckedException은 '예기치 못한, 치명적인 문제'로 봐서 자동으로 롤백함.**

## Spring 에서의 예외의 종류

#### UnChecked Exception : 시스템 오류, 개발자의 실수로 인식
- 롤백을 한다.
- (예: `NullPointerException`, `IllegalArgumentException`, `IndexOutOfBoundsException` 등)
#### Checked Exception : 정상적인 예외의 흐름으로 인식
- 롤백을 하지 않는다. 
- (예: `IOException`, `SQLException`, `FileNotFoundException` 등)

**예외적으로 롤백을 강제하거나 무시하고 싶을 때**  
`@Transactional(rollbackFor = ..., noRollbackFor = ...)` 속성 사용


## 1. **Checked Exception**

###  특징:

- **컴파일러가 예외 처리를 강제함**
    - `try-catch`로 처리하거나
    - `throws`로 던져야 함
- 예외를 **개발자가 예측하고 대비하라는 의미**
    
###  예시 상황:

- 파일이 없을 수 있다 → `FileNotFoundException`
- 네트워크가 끊길 수 있다 → `IOException`
- DB 접근이 실패할 수 있다 → `SQLException`
## **Unchecked Exception**

###  특징:
- **컴파일러가 예외 처리를 강제하지 않음**
- 주로 **프로그래머의 실수**, **논리 오류**를 의미
- 예외 처리를 안 해도 컴파일은 되지만, 런타임에 터짐
    
###  예시 상황:

- null 참조 → `NullPointerException`
- 잘못된 인덱스 → `IndexOutOfBoundsException`
- 잘못된 값 전달 → `IllegalArgumentException`

## 왜 **CheckedException**만 예외 처리를 강제할까?

#### 예외가 "예상 가능"하고, 개발자가 "꼭 처리해야만" 하는 상황이기 때문!**

## 그런데 왜 **UncheckedException**은 강제 안 할까?

#### **예외가 예측 불가능하고, 개발자의 실수로 발생하는 경우가 많기 때문!**
